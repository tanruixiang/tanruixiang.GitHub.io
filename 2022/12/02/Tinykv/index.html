Tinykv总结

本科的时候做过6.824，不过因为过去比较久了，加上6.824还是跟ETCD比还是比较玩具的，当然Tinykv也算是个大号玩具，不过毕竟代码结构和ETCD类似，所以还是和真实系统相近，所以准备做一做Tinykv复习复习。

总体难度3>2>4>1。2,3都有随机测试，不好复现，再加上要熟悉tinykv这套代码(应该是从tikv扒下来的)，调用链比较长，所以难度比较大，这里建议多参考别人的文章或者博客。其他都是线性测试，单步就能搞定。

Lab1

Lab1就是个热身，根据文档调badger包即可，这里没什么可讲的。badger包底层用的是LSM-Tree，没接触过的可以了解一下，看过levelDB或者是RocksDB的调包通过即可。

Lab2

Lab2就进入实现raft的阶段了，和6.824不同的是，6.824比较自由，接口什么的都是自己定，并且需要自己加锁防止race。而tinykv的raft接口设定是参照etcd设计的，不需要加锁。

lab2a

lab2a就是实现基本的raft操作，这里的测试都是线性的，并且基本和etcd的一模一样，强烈建议直接参考etcd来写。etcd除了多了一点优化和Message的名字不一样以外，其他几乎一模一样。这里强烈建议参考https://www.codedump.info/post/20180922-etcd-raft/，这篇文件里也有带注释的etcd代码，直接扒下来丢到tinykv上即可。

在这一节中，我们需要实现大部分raft模块的接口，代码量还是比较大的。一共有三个模块RawNode,Raft,RaftLog。这三个模块的定义和etcd一致。

  • RawNode:充当上层模块和raft模块交互的中间层，传递上层模块的信息给raft，比如各种各样的Msg。同时把raft的信息传递给上层模块,主要是各种日志的同步状态，比如向上传递已经同步可以apply的日志项。
  • Raft:最底层的raft模块，这里和6.824最大的不同是不需要上各种锁，因为这里的raft是靠上层Tick()驱动的，不会存在并发的情况。
  • RaftLog:管理日志存储相关操作，维护了一些日志相关的元数据。

如图所示，Raft和RaftLog共同维护Raft算法所要求的维护的一些信息。RawNode通过tick和step驱动Raft，而RawNode也会从Raft取出相应的数据，例如已经同步成功的log等放入到Ready，送入到上层状态中，而上层状态机就根据这个Ready来执行相应的操作。

RaftLog

// RaftLog manage the log entries, its struct look like:
//
//	snapshot/first.....applied....committed....stabled.....last
//	--------|------------------------------------------------|
//	                          log entries
//

RaftLog主要是要理解代码里注释中各个字段的含义。

  • Snapshot:只可以拿到Snapshot里最后一条日志的index和term。
  • stabled:这之前的是已经被持久化的日志
  • committed:这之前是已经被同步完的日志
  • applied是:这之前是已经应用了的日志

持久化日志是在上层执行的，把没有持久化的日志放到ready中，上层会把这些日志持久化起来。这里注意日志同步的时候有可能会改变stabled，因为stabled>=committed。
持久化的日志和没有持久化的日志共同构成了log entries。

Raft

Raft主要是实现各种Msg。可以看到pb中一共定义了13种消息类型。

const (
	// 'MessageType_MsgHup' is a local message used for election. If an election timeout happened,
	// the node should pass 'MessageType_MsgHup' to its Step method and start a new election.
	MessageType_MsgHup MessageType = 0
	// 'MessageType_MsgBeat' is a local message that signals the leader to send a heartbeat
	// of the 'MessageType_MsgHeartbeat' type to its followers.
	MessageType_MsgBeat MessageType = 1
	// 'MessageType_MsgPropose' is a local message that proposes to append data to the leader's log entries.
	MessageType_MsgPropose MessageType = 2
	// 'MessageType_MsgAppend' contains log entries to replicate.
	MessageType_MsgAppend MessageType = 3
	// 'MessageType_MsgAppendResponse' is response to log replication request('MessageType_MsgAppend').
	MessageType_MsgAppendResponse MessageType = 4
	// 'MessageType_MsgRequestVote' requests votes for election.
	MessageType_MsgRequestVote MessageType = 5
	// 'MessageType_MsgRequestVoteResponse' contains responses from voting request.
	MessageType_MsgRequestVoteResponse MessageType = 6
	// 'MessageType_MsgSnapshot' requests to install a snapshot message.
	MessageType_MsgSnapshot MessageType = 7
	// 'MessageType_MsgHeartbeat' sends heartbeat from leader to its followers.
	MessageType_MsgHeartbeat MessageType = 8
	// 'MessageType_MsgHeartbeatResponse' is a response to 'MessageType_MsgHeartbeat'.
	MessageType_MsgHeartbeatResponse MessageType = 9
	// 'MessageType_MsgTransferLeader' requests the leader to transfer its leadership.
	MessageType_MsgTransferLeader MessageType = 11
	// 'MessageType_MsgTimeoutNow' send from the leader to the leadership transfer target, to let
	// the transfer target timeout immediately and start a new election.
	MessageType_MsgTimeoutNow MessageType = 12
)
MessageType_MsgHup

本地消息,当tick选举时间超时的时候出触发该消息。需要检查自己是否在集群中。这里注意选举时间需要随机化。

MessageType_MsgBeat

本地消息，当tick心跳时间超时的时候出触发该消息。收到该消息的leader会向进群中的其他节点发送心跳包。

MessageType_MsgPropose

本地消息，上层需要proposal消息的时候会向raft传递该消息，收到该消息的leader会把信息append到自己的raftlog中，并且向集群中的其他节点同步自己的日志。

lab2b

lab2b开始就和之前的难度差距比较大了。个人认为2b的难点一个是Tinykv是从TiKV搬过来的，所以调用链比较长，需要搞清楚流程。第二个就是开始使用不稳定测试了，Bug更难找了。
首先需要理解文档中的概念。

  • Peer:Store里有多个Region，每个Region在当前Store下的所有peer共用底层存储，在TinyKV里是badger,现在比较流行的KV数据库一般是RocksDB
  • Store:节点，包含了多个Region。
  • Region:一个Region就是Raft集群，一个Region有多个Peer。Region是跨节点概念的。
    和Raft的Rawnode交互流程如下:
  1. Client发起请求。
  2. 在peer_msg_handler.go中把命令传递给Raft，例如proposal一个entry。
  3. Raft进行完相关操作后，使用RawNode接口会得到一个Ready结构。
  4. 根据Ready结构执行相应操作，如持久化日志，应用日志到状态机，应用snapshot等。这里会使用peer_storage.go的接口。
  5. 执行成功后向Client回包。

根据流程，我们需要实现的就是peer_msg_handler.go和peer_storage.go两个文件。

peer_msg_handler

peer_msg_handler需要实现两个函数proposeRaftCommand和HandleRaftReady。分别是用于proposal和apply的。

proposeRaftCommand

如果peerMsgHandler收到了MsgTypeRaftCmd消息，就会调用proposeRaftCommand，2b只涉及普通的Requests。只需要调用d.RaftGroup.Propose把Entry交给raft。在2b中,关于是否满足proposal的要求，在preProposeRaftCommand中已经实现了，比较简单。

HandleRaftReady

这里需要对raft层传上来的Ready进行相应的操作，具体有以下几个步骤。

  1. 首选要确认是否有待处理的Ready。
  2. 使用d.peerStorage.SaveReadyState保存有关snapshpt的信息。
  3. 使用d.peerStorage.SaveReadyState返回的ApplySnapResult更新region的相关信息。
  4. 发送ready中待发送的消息给别的peer。这里建议先发送，不然可能因为commit没发送就销毁自己导致测试卡住(当然这是lab3的测试了，同时lab3会针对这种这种情况做出优化，实际上优化后先后顺序基本没影响)
  5. 执行ready.CommittedEntries中的已经同步到大多数的日志。这里涉及apply每个entry,需要写入到对应的地方，实验文档里有相关介绍。(一开始我就上了batch write，导致后面lab3出现了问题，所以我建议直接一条一条写，后续再优化。)这里有个注意点：2b虽然没有实现snapshot，但是测试会有相关的request，需要使用d.proposals[0].cb.Txn = d.peerStorage.Engines.Kv.NewTransaction(false)来创建Txn,因为测试会使用Txn确认信息。
  6. d.RaftGroup.Advance(ready)驱动raft更新相关元数据。

peer_storage

这里主要是实现存储相关的操作。为peer_msg_handler提供存储接口。
涉及两个方法SaveReadyState,ApplySnapshot(2c使用)。2b第二个不需要实现。主要流程如下：

  1. 判断ready是否有snapshot，根据snapshot更新相关的元数据。
  2. 把ready.Entries需要持久化的entry持久化起来，写入到存储中。
  3. 更新ps.raftState.HardState。
  4. 使用raftWB.SetMeta(meta.RaftStateKey(ps.region.GetId()), ps.raftState)把需要持久化的状态写入到存储中。

lab2c

2c就是实现snapshot的过程了，这里调用链比较长，需要搞清楚生成snapshot和发送snapshot的流程。
可以参考https://asktug.com/t/topic/273859/2，说得比较清楚。

日志压缩过程

  1. peerMsgHandler收到MsgTypeTick消息，触发onTick()
  2. onTick中触发onRaftGCLogTick,该函数检查raft的日志是否已经达到了需要GC的程度，如果需要GC，那么就propse一个newCompactLogRequest给raft。
  3. 等待GC的entry同步后，就会在apply的时候开始生成snapshot。这个时候需要使用d.ScheduleCompactLog来产生一个GC TASK。同时需要更新applyState。
  4. raftlog_gc.go中raftLogGCTaskHandler收到task开始执行具体的GC操作。(需要注意，此刻是没有生成快照的，快照生成是由raft驱动的。可以说是因为日志被GC了，导致需要生成快照发送。)

快照生成和发送过程

  1. 当follower需要的entry已经被GC时。此时就会触发快照生成，从而进一步发送快照。具体是由r.RaftLog.storage.Snapshot()发生的。
  2. 快照生成具体是在peer_storage.go中的Snapshot,也就是raft调用的函数。这里会将ps.snapState修改成SnapState_Generating,并且发送一个RegionTaskGen任务。
  3. region_tack.go中收到task使用handleGen生成快照,调用了doSnapshot。这里主要是使用SnapManager进行相关操作。主要，快照的Index和Term是在getAppliedIdxTermForSnapshot获取的，主要是通过我们GC时候持久化的applyState.TruncatedState.Index和applyState.TruncatedState.Index。
  4. HandleRaftReady中使用send发送日志到其他的raftstore,这里知道是发送snapshot，所以会由snap_runner.go分块异步发送，具体细节对于完成Lab来说不需要了解。

快照接收过程

  1. 首先将快照的index和自己的commit对于，判断是否合法。
  2. 删除entries,更新committed stabled applied等元数据。并且设置l.pendingSnapshot = s。
  3. ready会把l.pendingSnapshot送给HandleRaftReady，用到上面提到的d.peerStorage.SaveReadyState来更新raftState和applyState。并且保存snapshot中的相关数据，这里是通过生成一个RegionTaskApply的task实现的。保存成功后，需要把各个state和region持久化。

lab3

lab3a

3a是给raft加上leaderTransfer和配置更改功能。因为都是线性测试，再加上配置变更都是以一个节点为单位的，和raft原文不同，所以比较简单。
leaderTransfer具体流程如下所示：

  1. 上层收到AdminCmdType_TransferLeader,调用d.RaftGroup.TransferLeader() 发送MsgTransferLeader给raft（3B中需要实现）
  2. 收到leaderTransfer后，会把r.leadTransferee 设置为需要转移的节点(需要保证当前节点是leader,并且设置后需要拒绝后续的所有proposal)。
  3. 根据Prs判断目标节点的日志是否和本节点一样新
  4. 如果一样新直接发送MsgTimeoutNow,目标节点收到MsgTimeoutNow后立刻发起选举，此时他有极大的概率选上leader。
  5. 如果不一样新，那么会立即向目标节点发送最新的日志(注意转移也有一个时间范围，超时了就放弃转移)。
    3a需要注意的有:
  6. 加入和删除节点都需要判断当前的配置是否存在该节点。
  7. leaderTransfer需要设置超时时间，否则可能出现转移一直卡死的情况。这里我的实现是直接和选举时间合一起了。
  8. leaderTransfer的时候需要拒绝proposal。
  9. 除了leaderTransfer以外，还需要维护r.PendingConfIndex,为了保证配置是整个raft集群以增或减一个节点为进度推进的，当前如果有entry是关于配置变化的，那么后面如果存在配置变化的entry需要忽略。这里也可以参考r.PendingConfIndex的注释。
  10. 配置变更后，需要重新计算commit，因为可能由于配置变更导致当前的日志已经满足了commit的条件。

lab3b

LeaderTransfer

如上所述。

ConfChange

这里就是给上层加上ConfChange和add/remove node的操作。感觉也算是坑比较多的一节了，具体可以看下面总结的坑点。需要注意的就是由于加了配置变化，每个命令都要检查Epoch和key。同时可能会存在超时的情况，有可能是raft同步太慢的原因，可以加一点优化，比如我就对appendentries加了hintindex的优化，超时现象就没出现过了。
具体流程如下：

  1. 收到AdminCmdType_ChangePeer,使用d.RaftGroup.ProposeConfChange,proposal变更entry。
  2. 由于proposal的entry类型是EntryType_EntryConfChange,需要保证不能同时存在多个未提交的EntryType_EntryConfChange。其他的方面在raft中和一个普通的entry完全一致
  3. 上层准备执行ConfChange时，首先检查CheckRegionEpoch。
  4. 根据是AddNode还是RemoveNode执行相应操作，首先需要检查当前peers中是否包括要执行的节点。增加.RegionEpoch.ConfVer，更新d.ctx.storeMeta.regions,根据配置使用*PeerCache插入或删除PeerCache。最后需要调用d.RaftGroup.ApplyConfChange对raft进行修改。
  5. 如果当前是leader，需要给pd发送心跳d.HeartbeatScheduler。

split

split应该是整个实验最难的一部分了。split解决的是一个region管理了过多的key，造成对单个region访问量过高导致并发量上不去的问题。当region的值大于一个阈值的时候，就会触发split操作。具体流程如下。

  • peer_mag_handler.go中onTick调用onSplitRegionCheckTick给split_checker发送一个splitcheck任务。

  • split_checker.go收到任务后，在splitCheck中检查是否分裂，检查的依据就是当前region的大小是否超过阈值，超过的话就会发送一个message.MsgTypeSplitRegion的消息给region。

  • region收到onPrepareSplitRegion后，会触发一个SchedulerAskSplitTask任务送到SchedulerTaskHandler

  • scheduler_task.go收到任务后在onAskSplit发送一个adminRequest。

  • 接下来就是我们写的代码部分了，和之前的命令类似，分别是proposal和apply部分。
    proposal我用CheckKeyInRegion检查了一下splitKey，其他就没有需要注意的了。

  • 主要是apply过程。过程如下

    • 检查部分，分别检查请求的epoch和splitkey 
      CheckRegionEpoch
CheckKeyInRegion
    • 更新storeMeta元数据，这里记得上锁。split所分配的新region的ID和PeerId已经由PD分配好了，这里主要是把这些ID填入到对应的数据结构中。主要涉及新建region和Peers。
    • 新建完后需要更新storeMeta中ID到region的映射以及key范围到region的映射，分别指的是regionRanges和regions。
    • 同时需要把两个region使用WriteRegionState持久化起来。
    • 创建对应的peer。在路由表router中注册Peer。
    • 最后如果当前是leader，那么就向PD发送心跳d.HeartbeatScheduler。
3b坑点总结

这里主要把本文中3b中上述提到的一些注意点合并起来了。3b是难度最大的一部分，写的时候错误甚至是跑几百次才复现一次，建议多参考别人的文档，我后面发现我遇到的很多问题都是共性的，已经有别人总结过了。

  1. 考虑只有两个节点的情况，此时如果Remove leader，leader收到follow的消息后更新了CommitID,然后自我销毁，如果此时follower没有收到Leader更新CommitID的包，这个时候follower永远不会在自己的配置中国删除leader。我的解决方案方案比较简单，当proposal的时候碰到这种情况时，直接触发transferleader。当然这样可能会导致time out问题，这里在raft里appendentries使用了hintIndex的方式后，就基本不会碰到time out问题了。
  2. apply的时候如果自己销毁了自己，在外层需要判断d.stopped，不然会写入脏数据，导致下次重启peer的时候检查出现问题。
  3. 配置变更后，需要尝试更新commit。
  4. 每次raft收到TransferLeader的包时，都需要根据Match发送entries或是Timeout,否则如果每次都需要等到TransferLeader超时，然后再收到上层的TransferLeader请求执行以上操作，很有可能会导致请求超时挂掉测试。
  5. 除了leaderTransfer以外，还需要维护r.PendingConfIndex,为了保证配置是整个raft集群以增或减一个节点为进度推进的，当前如果有entry是关于配置变化的，那么后面如果存在配置变化的entry需要忽略。这里也可以参考r.PendingConfIndex的注释。
  6. no region问题：这里就是pd处没有及时更新region信息，参考https://github.com/Smith-Cruise/TinyKV-White-Paper/blob/main/Project3-MultiRaftKV.md,加了一个`notifyHeartbeatScheduler`就好了。
  7. 值不匹配问题：由于我一开始就做了batch write的优化，导致整个写入流程有点乱。这里建议一开始就采用一条一条写入数据库的方法，先把测试用例过了，如果再出现超时的问题再用batch write的优化。
  8. apply的时候记得检查CheckRegionEpoch,CheckKeyInRegion。
